# 应用列表页顶部与分页交互设计

日期：2026-04-18

## 目标

统一 `MainActivity` 列表页顶部搜索区、分页结构与分页切换方式，确保实现结果与本轮视觉确认一致，不再在实现阶段反复猜测 UI 细节。

本设计只覆盖以下内容：

- 顶部搜索栏的结构
- 顶部外侧设置按钮与搜索框内筛选按钮的关系
- “全部应用 / 已配置应用” 两分页结构
- 通过左右滑动切换分页的交互语义

本设计不覆盖：

- 具体筛选项内容
- 设置页内容
- 列表数据加载、缓存、并发刷新等底层实现
- 底部导航或其它新入口

## 已确认的界面结构

### 顶部栏

当前态顶部采用三段结构，但视觉上只保留两块主元素：

1. 左侧主元素是搜索框
2. 右侧主元素是设置按钮
3. 筛选按钮不作为独立外部按钮存在，而是内嵌在搜索框内部右侧

搜索框内部结构固定为：

- 左侧：搜索 icon
- 中间：搜索提示文案或输入内容
- 右侧：筛选 icon

因此，顶部从左到右的实际可见顺序为：

`[搜索 icon + 搜索文本 + 筛选 icon] [设置按钮]`

约束：

- 搜索 icon 必须位于搜索框内部左侧
- 搜索框左边不能额外留一个独立“搜索按钮位”
- 筛选 icon 必须位于搜索框内部右侧
- 设置按钮必须继续保留在搜索框外侧右边
- 搜索框高度与已确认目标态保持一致，不再额外压矮或抬高

### 分页栏

顶部 tab 区域只保留两个分页：

- `全部应用`
- `已配置应用`

不再保留第三个分页。

视觉规则：

- 当前选中分页使用现有高亮风格
- 未选中分页保持现有弱化风格
- 页签顺序固定为左 `全部应用`、右 `已配置应用`

## 分页切换交互

分页切换支持两种方式：

1. 点击页签切换
2. 在列表区域直接左右滑动切换分页

关键约束：

- “左右滑动切换分页”是交互能力，不应额外引入一个肉眼可见的包裹容器边框
- 用户看到的仍然应该是普通列表区，而不是“列表被塞进一个新的滑动框”
- 因此实现上可以使用横向分页容器，但视觉上必须与当前普通列表保持一致

交互语义：

- 默认进入 `全部应用`
- 向左滑可进入 `已配置应用`
- 向右滑可返回 `全部应用`
- 点击 tab 与滑动分页必须保持状态同步

## 列表项展示层级

本轮确认没有改动列表项的主结构，继续维持当前态的三层信息：

- 第一行：应用名
- 第二行：包名
- 第三行：状态信息

同时保留：

- 左侧应用 icon
- 右侧进入详情的箭头

本设计没有把列表项改回“版本号优先”的目标态信息层级；当前实现阶段应先严格对齐已经确认的顶部与分页结构，不额外扩大改动范围。

## 实现边界

实现阶段允许：

- 复用现有 bottom sheet / settings 入口规范
- 使用现有 tab 样式语言
- 在不改变视觉层级的前提下，为列表区增加横向分页能力

实现阶段不允许：

- 把搜索 icon 再拆成搜索框外独立按钮
- 把筛选 icon 再移回搜索框外
- 去掉设置按钮
- 通过新增明显边框容器来表现可滑动分页
- 把两分页再次扩回三分页

## 验收标准

满足以下条件即可视为实现符合本设计：

1. 顶部右侧同时存在“搜索框内筛选 icon”和“搜索框外设置按钮”。
2. 搜索框左侧内部存在搜索 icon，且搜索框左边没有独立按钮空位。
3. 分页只有 `全部应用` 与 `已配置应用` 两项。
4. 点击 tab 可以切页。
5. 在列表区域左右滑动可以切页。
6. 列表区视觉上仍像普通列表，没有额外显眼包裹容器。

## 备注

这份设计是基于本轮浏览器对比稿 `list-compare-v7` 确认下来的结构结论。后续如果要继续讨论列表项信息层级、筛选项内容或加载性能，应作为下一轮独立设计处理。
